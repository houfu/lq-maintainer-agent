# PRD Research Report: LQ Maintainer Agent

Synthesized from 12 research dimensions covering plugin mechanics, headless services, commercial AI reviewers, pre-LLM OSS automation, LLM triage tools, prompt injection, eval harnesses, GitHub security, automation bias, PR untangling, dependency-update triage, and model lifecycle. Evaluated against the design doc at `/Users/houfu/Projects/lq-ai-maintainer/docs/design/lq-maintainer-agent-design-v0.5.1.md`.

---

## 1. Verdict: is this workable?

**Yes — workable, with high confidence, conditional on a defined set of corrections.** All 12 dimensions returned "workable" or "workable-with-changes"; none found a structural flaw requiring redesign. The corrections are numerous but they are refinements to mechanisms, claims, and rule content — the architecture (rules-as-data, thin skills, receipt-as-shared-state, human-gated writes, lane routing, eval-gated rules changes) survives contact with the evidence everywhere it was tested.

**Strongest evidence for:**

- **Every load-bearing mechanism has shipped precedent.** The exact end-to-end UX — install a plugin, run a review skill inside a clone, fetch via read-only `gh`, post via `gh pr comment` — is Anthropic's own first-party pattern in the [official code-review plugin](https://github.com/anthropics/claude-plugins-official/blob/main/plugins/code-review/commands/code-review.md). The update-in-place living receipt is PR-Agent's default (`persistent_comment=true`, [docs](https://docs.pr-agent.ai/tools/review/)); the never-approve posture is how [Copilot code review](https://docs.github.com/en/copilot/how-tos/use-copilot-agents/request-a-code-review/use-code-review) ships; the hidden-comment locator + resume-state pattern is exactly what users of claude-code-action are demanding ([issue #419](https://github.com/anthropics/claude-code-action/issues/419)) and what the Uniswap fork implements. The pre-LLM lineage is even deeper: rules-as-reviewable-data is [OWNERS files](https://www.kubernetes.dev/docs/guide/owners/) and [triagebot.toml](https://forge.rust-lang.org/triagebot/index.html); the merge-commit audit trailer is homu's `Auto merge of #N, r=reviewer` ([bors](https://bors.rust-lang.org/help)); rollup lanes are risk-based routing. Anthropic dogfoods the PRD's likely M4 landing zone in its own [claude-issue-triage.yml](https://github.com/anthropics/claude-code/actions/workflows/claude-issue-triage.yml).
- **The design directly answers the field's dominant complaints.** Signal-to-noise is the #1 documented grievance against every incumbent AI reviewer ([HN](https://news.ycombinator.com/item?id=46766961), [RedMonk on "AI slopageddon"](https://redmonk.com/kholterhoff/2026/02/03/ai-slopageddon-and-the-oss-maintainers/)), and the one-living-receipt + human-posts-everything design is more conservative than nearly anything deployed. Post-backlash community norms (the [Matplotlib/OpenClaw incident](https://www.theregister.com/2026/02/12/ai_bot_developer_rejected_pull_request/), [langchain's dosubot complaints](https://github.com/langchain-ai/langchain/discussions/25153), [curl's bug-bounty collapse](https://socket.dev/blog/curl-shuts-down-bug-bounty-program-after-flood-of-ai-slop-reports)) landed exactly where the PRD starts: draft-then-human-approve, report-only, human accountable for every public output. Mitchell Hashimoto's Ghostty workflow and Automattic's Gutenberg triage agent independently converged on the same shape ([SpecStory interview](https://newsletter.specstory.com/p/mitchell-hashimoto-on-the-ai-assisted), [nomad.blog](https://nomad.blog/2025/10/08/ai-triage/)).
- **The genuine novelties are defensible.** No surveyed product does attention routing, salvage decomposition, or anchoring against governing docs; no surveyed project publicly discloses its maintainer-side AI triage rubric. Those are real differentiators, not reinventions.

**Strongest evidence against (all survivable, none fatal):**

1. **The M0 safety floor overstates what it enforces.** The hook block-list is bypassable via at least four documented mechanisms — the ~50-subcommand deny-rule fallback ([Adversa](https://adversa.ai/blog/claude-code-security-bypass-deny-rules-disabled/)), Edit/Write self-modifying the hook/settings files ([issue #11226](https://github.com/anthropics/claude-code/issues/11226)), env-var prefixing ([#31558](https://github.com/anthropics/claude-code/issues/31558)), and `gh api`/GraphQL mutations that never match the blocked strings ([GraphQL mutations](https://docs.github.com/en/graphql/reference/mutations)) — plus hooks fail open under `--dangerously-skip-permissions` ([#20946](https://github.com/anthropics/claude-code/issues/20946)). The M0 exit criterion ("no Claude Code session can merge, approve, close, push…") is not achievable at the hook layer alone.
2. **The reproducibility claim is false as written.** Anthropic retires models ~12–14 months after release with 60 days' notice and no post-retirement access commitment ([deprecations](https://platform.claude.com/docs/en/about-claude/model-deprecations)); even a served pinned snapshot is not bitwise-deterministic ([Anthropic postmortem](https://www.anthropic.com/engineering/a-postmortem-of-three-recent-issues), [Thinking Machines](https://thinkingmachines.ai/blog/defeating-nondeterminism-in-llm-inference/)); and the receipt doesn't even record the model — which on subscription plans auto-switches by quota and safety fallback ([#16875](https://github.com/anthropics/claude-code/issues/16875)). "Re-run agent vX against head Y and canon Z" must become "auditable," not "reproducible."
3. **The human gate will erode.** A directly on-point longitudinal study of 11,429 human reviews of AI-agent PRs found approval rates rising +14.5pp and comment effort falling 22% with experience — "reflexive habituation, not trust calibration" ([arXiv:2606.22721](https://arxiv.org/abs/2606.22721)). And 45 real humans personally clicked merge on the malicious axios bump within hours ([GitGuardian](https://blog.gitguardian.com/renovate-dependabot-the-new-malware-delivery-system/)) — the PRD's fast-lane flow exactly. "Human click required" is necessary but not sufficient; the design must instrument whether the deciding stays real.
4. **The fast lane, as specified, re-derives with an LLM a judgment the dependency ecosystem makes deterministically** — and misses the one control with the best evidence (a release-age cooldown), while assigning the typosquat threat to the wrong artifact (typosquats arrive as *added* package names, not bumps).
5. **The injection posture is above industry baseline but leans on judgment where the strongest practice is layered** — it lacks unicode normalization (invisible payloads defeat "quote it as a finding"), an explicit rule for agent-instruction files in the diff (the highest-success vector: 83–100% in [GitInject](https://arxiv.org/html/2606.09935v1)), and a fix for the receipt footer as a cross-session re-injection channel. Anthropic's own triage workflow — the PRD's closest model — was successfully exploited ([Flatt Security](https://flatt.tech/research/posts/poisoning-claude-code-one-github-issue-to-break-the-supply-chain/)).

**Conditions under which it works:** (a) the enforcement boundary moves server-side (GitHub rulesets/branch protection + token permissions), with hooks demoted to defense-in-depth; (b) the fast lane is restructured as deterministic checks (cooldown, OSV lookup, no-new-names, verified bot identity) with the LLM as anomaly-spotter; (c) receipts/trailers gain a fourth pinned field (served model ID) and "reproducible" is reworded; (d) the injection rules gain normalization, the instruction-file rule, and a structured-fields-only footer; (e) the human gate gains cheap telemetry and canaries. All five fit the existing architecture.

## 2. How to work it

Ordered by importance. Where a PRD section is confirmed, it's marked ✅.

### 2.1 Rebuild the safety floor around server-side enforcement (§2.1, §10, §14-M0)

The true boundary a compromised or injected local session cannot bypass is GitHub-side: branch protection / rulesets (require PR, require code-owner review, dismiss stale approvals on push, block force-push, include administrators, restrict bypass actors) — plus, in M4, token permissions on the GitHub App (comment-write only, no merge capability at the credential level). Keep the hooks, but:

- Ship the block-list as **plugin hooks** (`hooks/hooks.json` PreToolUse), not a "settings mirror" — a plugin's bundled `settings.json` supports only `agent`/`subagentStatusLine` keys and cannot inject permission rules ([plugins reference](https://code.claude.com/docs/en/plugins-reference.md)). Document the known hook limits ([#20946](https://github.com/anthropics/claude-code/issues/20946), [#26923](https://github.com/anthropics/claude-code/issues/26923), stale-hook-path-until-`/reload-plugins`).
- Close the **`gh api` gap**: the block-list's string patterns miss `gh api -X PUT .../merge` and GraphQL mutations. Invert to an allow-list of read-only `gh` subcommands with `gh api` write methods denied by default.
- Pair CODEOWNERS on `/.claude/` with the branch-protection settings that make it binding ("require review from Code Owners" + "do not allow bypassing"), and note that CODEOWNERS never governs local working-tree edits ([OpenSSF](https://best.openssf.org/SCM-BestPractices/github/repository/code_review_not_limited_to_code_owners.html), [bypass catalog](https://github.com/jeremylong/Bypassing-Required-Reviews)).
- Add an **Allstar-style continuous verification job** — scheduled CI confirming via `gh api` that lq-ai's branch protection, CODEOWNERS routing, and hooks are still in place; one-time hardening leaves a temporary-disable window ([Allstar](https://github.com/ossf/allstar)).
- Reword the M0 exit criterion from "cannot" to "blocked at the hook layer and enforced server-side," and add an explicit rule in onboarding.md: never run triage sessions with `--dangerously-skip-permissions`.

### 2.2 Make the fast lane deterministic-first (§5, §15 q.5)

This is the single highest-value behavioral change. Restructure the merge-candidate conditions as:

1. Author verified as the dependabot/renovate **GitHub App identity** via API author-class (not display name or branch name).
2. Diff touches only manifest/lockfile paths ✅ (already in PRD).
3. Semver delta parsed as patch/minor on a ≥1.0.0 dependency (deterministic parse).
4. **No new package names** anywhere in the diff, including lockfile transitive churn — this one rule covers the typosquat/event-stream threat better than LLM name judgment ([event-stream post-mortem](https://snyk.io/blog/a-post-mortem-of-the-malicious-event-stream-backdoor/)).
5. Every changed name+version pair clears an **OSV batch lookup** against [ossf/malicious-packages](https://github.com/ossf/malicious-packages) MAL-/CVE advisories ([OpenSSF guide](https://openssf.org/blog/2026/05/20/detecting-malicious-packages-using-the-osv-api/)) — unauthenticated POSTs, compatible with the read-only posture.
6. **Release-age cooldown**: registry publish timestamp older than 7 days (Renovate recommends 14 for automerge; StepSecurity defaults 48h; malicious releases are typically pulled within 24–72h — [Renovate best practices](https://docs.renovatebot.com/upgrade-best-practices/), [cooldowns.dev](https://cooldowns.dev/), [StepSecurity](https://www.stepsecurity.io/blog/introducing-the-npm-package-cooldown-check)). The PRD currently has no time dimension at all, and this is the control that would have stopped the axios incident cold.
7. CI green ✅ — ideally with [dependency-review-action](https://github.com/actions/dependency-review-action) or osv-scanner as a required check in lq-ai itself (recommend upstream via §11), so the malware screen lives inside "CI green."

The LLM's residual role: verify "pure typo fix," anchor the bump to a real upstream release (§5 anchoring ✅), flag anomalies. **Answer q.5 now: advisory-driven majors never fast-lane** — Dependabot deliberately targets the minimum patched version to avoid majors ([GitHub docs](https://docs.github.com/en/code-security/concepts/supply-chain-security/about-dependabot-security-updates)); route them standard-lane with an "expedite" flag, and verify the advisory claim against GHSA/OSV, never the PR body — "urgent security fix" framing is itself a lane-promotion social-engineering vector. Add a receipt disclaimer that package *contents* were never inspected (the lockfile diff shows name+version+hash; ua-parser-js, chalk/debug, Shai-Hulud, and xz were all invisible at that layer — [CISA](https://www.cisa.gov/news-events/alerts/2025/09/23/widespread-supply-chain-compromise-impacting-npm-ecosystem), [Wiz](https://www.wiz.io/blog/widespread-npm-supply-chain-attack-breaking-down-impact-scope-across-debug-chalk), [Akamai on xz](https://www.akamai.com/blog/security-research/critical-linux-backdoor-xz-utils-discovered-what-to-know)).

### 2.3 Fix the plugin plumbing before M0 ships (§3.1, §3.3, §3.5)

- **Move `workspace/` out of the plugin tree.** Installed plugins are copied into a version-keyed ephemeral cache; docs say "do not write state here." Use `${CLAUDE_PLUGIN_DATA}` (persistent, survives updates) for the deep-dive cache ([plugins reference](https://code.claude.com/docs/en/plugins-reference.md)).
- **Correct the `allowed-tools` framing in §10**: it *grants* promptless use, it does not restrict — restriction requires deny rules/hooks ([skills docs](https://code.claude.com/docs/en/skills.md), [permissions](https://code.claude.com/docs/en/permissions)). The permission-gated posting flow works only if write commands are *omitted* from `allowed-tools` — and note that Anthropic's official code-review plugin allowlists `gh pr comment:*`, so a naive copy of its frontmatter would silently break the accountability model.
- Set `disable-model-invocation: true` on both skills (this is the documented mechanism for §3.3's explicit-invocation requirement ✅), reference rules via `${CLAUDE_PLUGIN_ROOT}` paths (the cwd is the lq-ai clone, not the plugin), and validate the namespaced invocation (`/lq-maintainer:triage`, not bare `/triage`).
- Versioning works exactly as assumed ✅ — set `version` in plugin.json, bump every release, third-party marketplaces don't auto-update by default ([discover-plugins](https://code.claude.com/docs/en/discover-plugins)). Record the plugin.json semver in receipts.
- **Context lifecycle caveat for batch `/triage`**: compaction keeps only ~5,000 tokens per invoked skill, so long batch sessions can lose rule text — fork a subagent per item or re-read the governing rule file per decision, or late-batch lane calls are made from summarized memory.
- Note the platform constraint for §9: plugin-shipped agents cannot carry hooks/mcpServers/permissionMode; the review team's read-only posture must come from `tools`/`disallowedTools` frontmatter.

### 2.4 Harden the injection posture with three concrete additions (§5, §8.4, §10)

- **Input normalization first**: NFKC + strip/reject Unicode Tags block (U+E0000–E007F), zero-width chars, and bidi overrides on all untrusted spans — PR bodies, diffs, commit messages, *filenames* ([Tenable TRA-2025-53](https://www.tenable.com/security/research/tra-2025-53)) — and prior receipt footers. Without it, "quote the injection as a finding" fails silently on invisible payloads ([promptfoo](https://www.promptfoo.dev/blog/invisible-unicode-threats/), [AWS guidance](https://aws.amazon.com/blogs/security/defending-llm-applications-against-unicode-character-smuggling/)).
- **Agent-instruction files in the diff are data, never instructions**: CLAUDE.md, AGENTS.md, `.claude/**`, copilot-instructions, and tool-config files (`.rubocop.yml`/`conftest.py` — the [CodeRabbit 1M-repo RCE class](https://www.propelcode.ai/blog/coderabbit-vulnerability-how-ai-code-review-security-flaw-exposed-1m-repositories)) get flagged as escalation-worthy findings, never loaded. This is the highest-success documented vector ([GitInject, 83–100%](https://arxiv.org/html/2606.09935v1)).
- **Constrain the footer to enumerated structured fields only** (lane enum, rule-id, SHAs, model, version, coverage booleans, finding-ids) — never free-text quoted contributor content. The footer is an HTML comment, the exact concealment channel Copilot Agent was exploited through ([Comment and Control](https://oddguan.com/blog/comment-and-control-prompt-injection-credential-theft-claude-code-gemini-cli-github-copilot/)); a quoted injection re-parsed on resume re-enters at elevated trust. Also verify the prior receipt's **author identity** before trusting its footer (the pre-LLM norm — state keyed to bot identity, never content), and adopt triagebot's rule that command/footer-like text inside code blocks and blockquotes is inert ([triagebot](https://forge.rust-lang.org/triagebot/index.html)).
- State explicitly in §10 that the load-bearing defense is the **permission architecture**, not the prompt — Anthropic reports ~1% residual attack success even on RL-hardened agents ([Anthropic](https://www.anthropic.com/research/prompt-injection-defenses)); frame against [Meta's Rule of Two](https://ai.meta.com/blog/practical-ai-agent-security/), which the M1–M3 design already satisfies by construction and M4 does not.

### 2.5 Rewrite reproducibility as auditability; pin the model (§3.4, §8.4, §8.5)

Add the **actually-served model ID** as a fourth pinned field (from `modelUsage` in headless JSON output; from session info interactively — [model-config](https://code.claude.com/docs/en/model-config)). Treat model migration as a rules-change-equivalent event: pin the CI eval model by full name, run the golden suite on old vs. new with a human-reviewed lane diff before switching, bump the agent version. Budget for a forced migration roughly annually (60-day notice, the shortest of the major providers). Reframe disputes: adjudicated against recorded reasoning + the eval suite, not literal re-execution — which becomes impossible ~12 months out and was never bitwise-faithful anyway. Add the two new open questions (model-migration policy; retired-model dispute procedure) to §15.

### 2.6 Instrument the human gate (§5, §8.4/§8.5, §13)

Cheap additions riding existing plumbing:

- Two footer/trailer telemetry fields — decision latency and human-delta (did the human change anything) — so the on-demand `git log` digest reports **override rate and approval velocity**, with a stated tripwire (sustained 0% override → adversarial audit of lane rules). No surveyed product does this; the PRD would be ahead of the market.
- A **canary regime**: occasionally seed one synthetic adversarial fixture (from `evals/fixtures/`) into a live digest and record catch/miss — the [Threat Image Projection](https://pmc.ncbi.nlm.nih.gov/articles/PMC8952858/) analogue.
- Note that confidence scores and rule citations — the PRD's chosen calibration displays — are the two mitigations the [Microsoft Research overreliance review](https://www.microsoft.com/en-us/research/publication/overreliance-on-ai-literature-review/) classifies as intuitive but unsupported; consider pairing fast-lane one-liners with one falsifiable cognitive-forcing fact ("diff touches only package-lock.json, 2 files, 14 lines") per [Buçinca et al.](https://arxiv.org/abs/2102.09692)
- Address permission-prompt habituation ([habituation collapses by the second identical exposure](https://dl.acm.org/doi/10.1145/2702123.2702322)): vary dialog content per item, and document why comment-writes must never enter the allowlist — "always allow" deletes the gate with one keystroke, and the PRD never mentions it.
- Flag the M2 dependency: "corrections become fixtures" assumes maintainers keep correcting; under rubber-stamping the stream dries up while looking healthy. Override-rate telemetry is its health metric.

### 2.7 Upgrade the eval harness with four cheap additions (§4)

The golden-mechanical + advisory-model-graded + corrections-flywheel design is state of the art ✅ ([Hamel Husain](https://hamel.dev/blog/posts/evals-faq/), [Anthropic agent evals](https://www.anthropic.com/engineering/demystifying-evals-for-ai-agents), [Inspect](https://inspect.aisi.org.uk/)). Add:

1. **k-trial repetition**: temperature 0 is not deterministic; grade the never-fast-lane invariant **pass^k** (fail on any trial), ordinary lanes by threshold. Single-run grading is flaky *and* falsely reassuring.
2. **Judge validation before advisory results are shown**: binary rubric, CoT-before-verdict, judge model distinct from the triage model (self-preference bias 10–25% — [survey](https://arxiv.org/html/2606.19544v1)), TPR/TNR against ~100+ maintainer labels. This converts q.6 into a measurable graduation bar.
3. **Negative fixtures** (escalation must NOT fire; a trivial PR MUST fast-lane) — one-sided suites drift toward escalate-everything, which destroys the tool's value.
4. **Fixture lifecycle**: record the canon SHA each golden outcome was adjudicated under; the canon-pin-advance PR must re-verify fixtures whose correct answer may have changed. Report a per-lane confusion matrix (the fast-lane false-positive cell is the safety number) and baseline diffs, Braintrust-style. Growth targets: ~20–50 fixtures by M2, 100+ by M3. Salvage golden outcomes must be acceptance *sets* — trained human raters fail to agree on ~11% of lines ([tangled-changes dataset](https://arxiv.org/abs/2011.06244)).

### 2.8 Demote salvage step 4; keep steps 1–3 as the required output (§6)

Detection and naming of concerns is the high-accuracy capability (tangling detection F1 ~0.88 — [arXiv](https://arxiv.org/html/2505.08263v1)); hunk assignment tops out at ~70% and degrades with size ([ColaUntangle](https://arxiv.org/html/2507.16395)), which is exactly the overreaching-PR regime, and no tool verifies splits compile. Ship the hunk map as an explicitly-unverified advisory draft with a mandatory receipt line ("proposed split not verified to compile or pass tests"), add cheap blocking mechanical checks (partition covers the whole diff; grep-based defined-here/used-there cross-reference), demote to file-level proposals above a size threshold, and default to **maintainer-performed splits** in contributor responses — handing a novice unverified rework is how contributions die ([abandonment study](https://dl.acm.org/doi/10.1145/3530785)). File issue-side splits as GitHub **sub-issues** (GA April 2025 — [GitHub changelog](https://github.blog/changelog/2025-04-09-evolving-github-issues-and-projects/)). The §9 multi-agent structure is validated: the strongest published untangler is itself explicit-dependency-agent + implicit-agent + reviewer.

### 2.9 Build order

M0 stands, with the reworked exit criterion, server-side rulesets, plugin-hook block-list, and a **spike acceptance test** that the hooks actually fire from a marketplace-installed plugin. M1–M3 stand. For M4, add **Option D** (Claude Code [Routines](https://code.claude.com/docs/en/routines)/Managed Agents) to the spike matrix — and likely reject it for publication runs because routines run under the maintainer's *personal* identity, violating visibly-agent-authored receipts; evaluate [claude-code-action](https://github.com/anthropics/claude-code-action) before any bespoke service (it installs plugins in CI and runs namespaced skills headlessly, validating §3.2's port-unchanged claim ✅). Budget on metered API pricing: Anthropic's terms bar claude.ai-subscription auth for SDK-powered services; measured costs bracket $0.04–$1.35/single-pass review vs. $15–25/multi-agent deep dive ([Kilo analysis](https://blog.kilo.ai/p/we-analyzed-how-much-kilo-code-reviewer), [Anthropic on X](https://x.com/claudeai/status/2031088175456903667)) — which quantifies q.7 at a $1–5 per-PR ceiling for M3 opt-in.

| **Pre-LLM: [Prow/OWNERS/Tide](https://docs.prow.k8s.io/docs/overview/), [bors/triagebot](https://forge.rust-lang.org/triagebot/index.html), [Mergify](https://docs.mergify.com/configuration/file-format/), [Allstar](https://github.com/ossf/allstar), [VS Code triage](https://github.com/microsoft/vscode/wiki/Automated-Issue-Triaging)** | Decade-tested governance automation | Rules as reviewed data files; `needs-triage` default state; human sign-off as auditable commands; reviewer identity baked into merge commits; label-keyed workflow state; bot permissions granted via reviewed PRs | PRD is *more* conservative than the field norm (k8s/Rust let the bot merge after human approval; PRD makes the human click) — defensible. **Gaps:** lane state invisible to GitHub search (add a tiny label mirror); no contributor `/hold` analog; no documented contest path. |
| **[Renovate merge confidence](https://docs.renovatebot.com/merge-confidence/) / Dependabot cooldown** | Deterministic bump-safety judgment | Age/adoption/pass-rate crowd signals; `minimumReleaseAge` 14 days for automerge; security updates bypass cooldowns (a known weakness) | **Gap:** the PRD's LLM cannot see crowd signals or package contents; the deterministic substitutes (age, OSV, deps.dev) must be named. |

The two things the PRD does that nothing surveyed does — **machine-readable footer as cross-session shared review state** (commercial tools keep state server-side) and **triage/lane-routing + salvage rather than review** — both look like genuine advantages, provided the footer hardening in §2.4 lands. The one place the PRD is *behind* the universal norm: until M4, receipts post under the maintainer's own identity with no bot badge — every incumbent posts under a distinct App identity, so add a visible attribution line ("Drafted by lq-maintainer-agent vX; reviewed and posted by @maintainer") per-artifact, not just in CONTRIBUTING.

## 4. Best practices the PRD overlooked

Deduplicated, priority order:

1. **Release-age cooldown for dependency updates.** The ecosystem's primary evidenced control ([Renovate](https://docs.renovatebot.com/key-concepts/minimum-release-age/), [Dependabot cooldown](https://docs.github.com/en/code-security/reference/supply-chain-security/dependabot-options-reference), pnpm/npm native support); malicious releases pulled within 24–72h ([analysis](https://christian-schneider.net/blog/dependency-cooldowns-supply-chain-defense/)). → Add ≥7-day age rule to `rules/lanes.md`; also recommend lq-ai adopt `dependabot.yml cooldown` upstream.
2. **Server-side rulesets as the enforcement spine, hooks as depth.** ([OpenSSF](https://best.openssf.org/SCM-BestPractices/github/repository/code_review_not_limited_to_code_owners.html), hook-bypass issues above.) → Rework §2.1/M0; add "dismiss stale approvals on push" (currently the force-push-invalidates-review logic is agent convention only).
3. **Unicode/ASCII-smuggling normalization** of all untrusted input, filenames included. ([promptfoo](https://www.promptfoo.dev/blog/invisible-unicode-threats/), [AWS](https://aws.amazon.com/blogs/security/defending-llm-applications-against-unicode-character-smuggling/)) → First step in `rules/injection-posture.md` + an eval fixture proving it runs.
4. **Agent-instruction files in the diff treated as findings, never instructions.** ([GitInject](https://arxiv.org/html/2606.09935v1), [GitHub's reviewer guidance](https://github.blog/ai-and-ml/generative-ai/agent-pull-requests-are-everywhere-heres-how-to-review-them/)) → Explicit rule in §5/§10; three new golden fixtures (invisible-unicode payload, CLAUDE.md-in-diff, tool-config RCE file).
5. **Served-model recording + migration runbook.** ([model deprecations](https://platform.claude.com/docs/en/about-claude/model-deprecations), [Braintrust workflow](https://www.braintrust.dev/docs/evaluate)) → Fourth pinned field; eval-gated migrations; archive per-model eval results as the substitute for replay.
6. **Human-gate instrumentation**: override-rate/velocity telemetry, live canaries, polymorphic permission prompts, no-"always-allow" rule. ([Habituation at the Gate](https://arxiv.org/abs/2606.22721), [warning-habituation fMRI work](https://dl.acm.org/doi/10.1145/2702123.2702322), [EU AI Act Art. 14(4)(b)](https://artificialintelligenceact.eu/article/14/) treats enabling automation-bias awareness as baseline oversight.) → §8.4/§8.5 fields + onboarding note.
7. **Edited comments notify nobody.** ([PR-Agent's `final_update_message`](https://docs.pr-agent.ai/tools/review/)) → Post a one-line "receipt updated" ping or document that resume-discovery is pull-based. Also **version the footer schema** (`<!-- lq-maintainer-agent:receipt:v1 -->`) — marker-format changes break persistent-comment lookup.
8. **A spam/AI-slop disposition.** The motivating crisis for peers ([curl](https://socket.dev/blog/curl-shuts-down-bug-bounty-program-after-flood-of-ai-slop-reports)); GitHub ships a conservative slop-detection workflow ([GitHub Models cookbook](https://github.blog/open-source/maintainers/how-github-models-can-help-open-source-maintainers-focus-on-what-matters/)). → Fifth disposition with an "only flag obvious slop" rule; also a policy for contributions from autonomous AI agents (author-class covers "external," not "non-human").
9. **Stale-sweep guardrails**: never stale items awaiting maintainer response; reactions/subscriptions count as interest; frozen/exempt marker; "stale ≠ resolved" requires evidence in drafted closes. (The [probot/stale backlash](https://github.com/pypa/virtualenv/issues/1311), [nostalebots.xyz](https://nostalebots.xyz/), langchain's dosubot complaints.) → Encode in `rules/` for §7.
10. **A finding-filtering stage + severity cap on receipts.** (Ellipsis architecture; every incumbent grew a severity dial.) → Dedup + confidence threshold + evidence-check between §9 subagent output and the receipt; cap findings with the remainder collapsed.
11. **Eval-harness mechanics**: pass^k on security fixtures, judge validation, negative fixtures, fixture re-adjudication on canon-pin advance, response caching, per-lane confusion matrix, nightly generated-adversarial tier ([promptfoo redteam](https://www.promptfoo.dev/docs/red-team/configuration/)), periodic grading audits (Anthropic documents a 42%→95% swing that was all grading bugs). → §4.2 additions per §2.7 above.
12. **Headless hygiene bundle for M4**: `settingSources` restricted so the lq-ai checkout's own `.claude/`/CLAUDE.md is never loaded; fork-PR secrets semantics stated as the *structural* reason triggers can't live in lq-ai ([pwn requests](https://securitylab.github.com/resources/github-actions-preventing-pwn-requests/), [checkout v7 hardening](https://github.blog/changelog/2026-06-18-safer-pull_request_target-defaults-for-github-actions-checkout/)); bot/actor filtering so the agent never ingests its own or dependabot's comments; per-author/per-PR run caps against cost-DoS; cron keepalive + dead-man's-switch (scheduled workflows silently disable after 60 commit-free days — [community discussion](https://github.com/orgs/community/discussions/156282)); Workload Identity Federation instead of a static ANTHROPIC_API_KEY (what Anthropic's own triage workflow uses); secret-scan agent output before posting; treat public run logs as an exfil channel, not free transparency (the [Flatt exploit](https://flatt.tech/research/posts/poisoning-claude-code-one-github-issue-to-break-the-supply-chain/) used them).
13. **Minor label/state hygiene**: a tiny `lane/*` + `needs-triage` label mirror so state is searchable; a contributor-visible hold/contest path; a published bot-behavior page (the [go.k8s.io/bot-commands](https://www.kubernetes.dev/docs/guide/issue-triage/) norm); cross-reference the emerging `Assisted-by:` trailer convention ([allthingsopen](https://allthingsopen.org/articles/open-source-ai-contributions-assisted-by-git-trailer-standard)) so §8.5 reads as part of a recognized standard; `userConfig` in plugin.json for per-maintainer settings; supply-chain scanning (Scorecard/zizmor) of the agent repo's own workflows.

## 5. Risks and open questions

**Risks, ranked:**

1. **Safety-floor overclaim** (M0). If the exit criterion ships as written, the project believes it has a boundary it doesn't. Mitigation is §2.1 above; residual risk after server-side enforcement is low.
2. **Fast-lane compromise window.** Demonstrated live (axios, 45 human clicks); closed almost entirely by the cooldown + OSV + no-new-names stack. Residual: compromised-maintainer releases older than the cooldown — covered only by the contents-not-inspected disclaimer and the human-only supply-chain-hygiene item.
3. **Receipt-footer re-injection and spoofing.** New attack surface the PRD itself introduces; closed by structured-fields-only + author-identity verification + normalization on re-parse.
4. **Human-gate erosion.** Predicted by strong evidence; not preventable, but detectable with the telemetry/canary additions. The uninstrumented version risks the receipt becoming what Ben Green calls a legitimizing artifact — proof a human clicked, not that a human decided ([Green](https://papers.ssrn.com/sol3/papers.cfm?abstract_id=3921216)).
5. **M4 removes the load-bearing defense.** Per-post human approval is what makes the M1–M3 injection posture hold; the headless plan currently has no equivalent. The Flatt exploit of Anthropic's own workflow is the case study. The spike deliverable must include the classifier layer, egress allowlist, and zero-secrets runner design, or M4 is a no-go.
6. **Forced model migration** (~annually, 60 days' notice) with no migration runbook — currently would land as unexplained judgment drift.
7. **Eval-suite blind spots**: single-run flakiness masking a 1-in-5 adversarial fast-laning; one-sided fixtures drifting rules toward escalate-everything (which quietly kills the tool's value for a part-time maintainer).
8. **Researcher disagreements to note**: dimension 1 recommends the plugin hooks as a "traveling backstop" while dimension 8 catalogs their bypasses — reconciled here as hooks-as-depth, rulesets-as-boundary; weight dimension 8, since it cites reproduced bypasses. Dimension 3 suggests optional inline suggested-change comments while the noise literature warns against comment proliferation — resolvable because each inline post is still human-gated and opt-in; try it only after M3 receipts prove quiet. And "public logs = transparency for free" (§12 Option A) is contradicted by three independent researchers (exfil channel, secret-masking unreliability, claude-code-action's own `show_full_output` default) — treat the receipt as the public artifact and suppress raw transcripts.

**Changes to §15's open questions:**

- **q.1 (committee mechanics)** — unchanged, but the access-control requirement is now stronger since packets also carry all sensitive review state.
- **q.2 (author-trust classes)** — answered by precedent: use a reviewed allowlist file (k8s org membership, Rust's team-database PR, Home Assistant CODEOWNERS all use reviewed data files); avoid "N merged PRs" heuristics. Additionally distinguish *non-human* authors, which author-class currently doesn't.
- **q.3 (soft trailer check)** — answer **yes, warn-only**; it maps onto accepted DCO warn-check practice, and homu's `r=reviewer` convention is the citation for §8.5.
- **q.4 (receipt placement)** — comment-first confirmed (the action's Checks permission is "not yet active"; its own pattern is an updated tracking comment); Bugbot's comment+status-check hybrid is the M4 upgrade path.
- **q.5 (advisory-driven majors)** — answered: **no**; expedited standard lane, advisory verified against GHSA/OSV.
- **q.6 (model-graded graduation)** — now measurable: judge TPR/TNR ≥ bar against maintainer labels for N consecutive cycles, with a distinct pinned judge model. For salvage specifically, graduation also requires acceptance-set-based grading.
- **q.7 (deep-dive budget)** — calibrated: market rate $1–5/PR for deep review (Bugbot $1.00–1.50 avg; managed multi-agent $15–25); set the opt-in ceiling in that band.
- **q.8 (digest scale)** — unchanged; defer stands.
- **New questions to add**: (9) model-migration policy — who approves, what eval delta blocks; (10) retired-model dispute procedure; (11) M4 injection posture without a human — classifier layer, egress, secrets design as go/no-go criteria; (12) footer schema versioning and the update-notification convention; (13) what run detail is public in headless mode (receipt yes, transcripts no).
